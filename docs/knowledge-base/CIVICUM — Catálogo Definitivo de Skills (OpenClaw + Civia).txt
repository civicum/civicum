# CIVICUM — Catálogo Definitivo de Skills (OpenClaw + Civia)  
**Versión:** 1.0 (Catálogo Maestro)  
**Propósito:** Maximizar automatización **segura y aislada** para:  
1) **Asistir a Civia** sin exponer OpenClaw a ciudadanos (solo por *artefactos internos*).  
2) **Gestionar Civicum post-lanzamiento** (operación nacional) con la menor carga diaria para ti.  
3) Implementar “auto-mejora” **segura** (medir → evaluar → proponer → aprobar), sin auto-update ni acciones destructivas autónomas.

---

## 0) Arquitectura operativa objetivo (resumen ejecutivo)

### 0.1 Separación absoluta de planos
- **Plano Ciudadano (Civia):** responde con tu stack client-first (3 pisos).  
- **Plano Operacional (OpenClaw):** interno, aislado, nunca atiende a ciudadanos.

### 0.2 Cloud-3 interno (OpenClaw)
- **OpenClaw PROD-OPS (24/7):** operación, seguridad, datos, contenido, runbooks, incidentes.  
- **OpenClaw DEV (on-demand):** vibe coding puro (PRs, tests, refactors, docs).  
- **Aegis (obligatorio):** proxy/firewall semántico con *denylist destructiva*, SSRF guard, allowlists, y políticas HITL.
- **Acceso remoto seguro:** Tailscale o Cloudflare Tunnel (sin puertos públicos).  
- **Volúmenes cifrados:** LUKS (config/workspace/keys).  
- **Pinning de versiones:** nada de “latest”; self-update deshabilitado.

---

## 1) Principios no negociables (enforced-by-design)

### 1.1 Regla de oro de autonomía
- **READ**: autónomo (con límites + auditoría).  
- **WRITE/UPDATE/DELETE**: **HITL obligatorio**.  
- **Acción destructiva o irreversible**: **doble aprobación (two-person rule) + break-glass**.

### 1.2 “Auto-mejora” permitida (sin riesgos)
OpenClaw puede:
- Medir desempeño (telemetría), detectar gaps, generar propuestas, preparar PRs/packs, correr evaluaciones y *sugerir* cambios.  
OpenClaw **NO** puede:
- Auto-actualizarse, rotar secretos sin aprobación, “purge all”, desactivar seguridad, borrar datos, abrir accesos públicos, ni ejecutar comandos destructivos autónomos.

### 1.3 Evidence-Only + Abstención Dura (para Civia y datos cívicos)
- Si no hay evidencia interna/validada: **abstenerse** o responder con guía general + “no puedo confirmar”.  
- Prohibido “inventar” plazos, artículos, costos, procedimientos, URLs oficiales.

---

## 2) Modelo de Skills: estándar, permisos, riesgos

### 2.1 Niveles de riesgo (R0–R4)
- **R0 Observación:** lectura pura (autónomo).  
- **R1 Diagnóstico:** lectura + análisis (autónomo).  
- **R2 Propuesta:** produce artefactos (PR/Issue/Pack Draft), no publica (HITL para merge/publish).  
- **R3 Acción controlada:** cambios reversibles (HITL).  
- **R4 Crítico:** irreversible / seguridad / secretos / migraciones (HITL + two-person + ventana).

### 2.2 Contrato de Skill (Skill Card)
Cada skill debe definir:
- **ID / Nombre / Dominio / Entorno (PROD-OPS|DEV) / Riesgo**  
- **Objetivo** + **Disparadores**  
- **Entradas** (schema) + **Salidas** (artefactos)  
- **Permisos mínimos** (tags Aegis)  
- **HITL** (cuándo y cómo apruebas)  
- **Safety checks** (allowlist/denylist, dry-run, límites)  
- **Auditoría** (eventos + evidencia + trace_id)  
- **Rollback** (si aplica) + **Tests** + **KPIs**

---

## 3) Puentes internos OpenClaw → Civia (sin conversación directa)

### Puente A — Knowledge Pack (el principal)
OpenClaw mantiene un **paquete versionado** con:
- Plantillas JSON (Piso 0), FAQs validadas, reglas de seguridad, disclaimers, y (opcional) índices de retrieval.  
- Publicación a Civicum mediante **endpoint admin interno** (HITL).

### Puente B — Runbooks & Operación
OpenClaw transforma incidentes/telemetría en:
- runbooks accionables, postmortems, checklist de releases, hardening.

### Puente C — Moderación asistida
- Sugiere priorización y revisión; jamás sanciona automáticamente.

## Dominio KP — Knowledge Pack (Civia mejora sin exponer OpenClaw)

### KP-001 Inventario canónico de contenido (R0, PROD-OPS)
- Objetivo: listar plantillas/FAQs/reglas/versiones/hashes/owners.
- Entradas: repo + storage de packs.
- Salidas: `kp_inventory.json`.
- Permisos: `fs:read`.
- Safety: checksum + firma.
- KPI: cobertura ≥ 99%.

### KP-002 Linter de plantillas JSON Piso 0 (R1, PROD-OPS)
- Objetivo: validar schema, placeholders, localización, tono.
- Salidas: reporte + fixes sugeridos (no aplica).
- Safety: bloquear campos peligrosos (HTML sin sanitizar).
- Tests: render tests.

### KP-003 Compilador de Knowledge Pack (R2, PROD-OPS)
- Objetivo: construir `kp_build_vX.Y.Z` con manifiesto.
- Salidas: artefacto + `manifest.json` + `artifact_hash` (SHA-256).
- HITL: requerido para publish.
- Safety: firma/hashes obligatorios. NO_OVERWRITE_POLICY: Falla si el artefacto destino ya existe. Requiere incremento de versión semántica.

### KP-004 Diff semántico entre packs (R1, PROD-OPS)
- Objetivo: detectar cambios “breaking” en respuestas.
- Salidas: `impact_report.md` (por categoría/módulo).
- Safety: marca “riesgo alto” si cambia disclaimers/leyes.

### KP-005 Generador de “golden answers” (R2, DEV)
- Objetivo: crear suite de regresión de respuestas (fixtures).
- Salidas: tests + casos canónicos.
- HITL: merge humano.
- KPI: regresiones detectadas antes del publish.

### KP-006 Validador Evidence-Only (R1, PROD-OPS)
- Objetivo: verificar que cada claim cite fuente interna válida.
- Salidas: lista de claims sin evidencia → *abstención obligatoria*.
- Safety: bloqueo de publish si falla.

### KP-007 Generador de disclaimers por tipo de consulta (R2, DEV)
- Objetivo: disclaimers consistentes (legal/financiero/salud/seguridad).
- Salidas: snippets + reglas de inserción.

### KP-008 Normalizador de tono Civia (R2, DEV)
- Objetivo: reescribir sin cambiar hechos.
- Safety: diff factual (no cambia números, plazos, requisitos).

### KP-009 Detector de contenido desactualizado (R1, PROD-OPS)

- Salidas: issues + prioridad.

### KP-010 Validador de links oficiales (R1, PROD-OPS)

### KP-011 Generador de FAQs (por módulo) (R2, DEV)

# 5) Cómo “incorporarlo” al desarrollo y a la gestión total post-lanzamiento

## 5.1 Flujo operativo diario (para quitarte carga)
- **Mañana (10 min):** OPS-013 (digest) + SEC-001 (seguridad) + FIN-001 (cuotas).  
- **Durante el día:** OpenClaw hace triage y te deja *colas listas* (OPS-003, SUP-001).  
- **Cierre:** KP-002 + DATA-014 (staleness) + EVAL-001 (telemetría).

## 5.2 Ciclo de mejora continua (auto-mejora segura)
1) **Medir**: EVAL-001 + EVAL-002.  
2) **Detectar gaps**: EVAL-005 (abstención), OPS-015 (problemas repetidos), SUP-002 (FAQ emergentes).  
3) **Proponer**: KP-011/KP-012, DATA-007, REL-001/REL-002, GOV-003.  
4) **Evaluar**: EVAL-004 + REL gates.  
5) **Aprobar**: HITL (tú) + two-person si crítico.  
6) **Publicar**: KP-017 (pack), DATA-012 (datos), REL-007 (deploy).  
7) **Rollback automático sugerido** (no ejecutado) si regresión: KP-018 / REL-008 / DATA rollback.

## 5.3 “Vibe coding puro” sin riesgo
- OpenClaw DEV solo:
  - lee repo → propone PRs → genera tests → deja todo listo.
- Tú decides:
  - merge, deploy, migraciones, cambios de seguridad.

7) Nota final (lo más importante)

Este catálogo está diseñado para que OpenClaw:

No “reemplace” a Civia (no conversa con ciudadanos).

Te reemplace a ti en el trabajo repetitivo: triage, reportes, propuestas, PRs, packs, runbooks, checks, auditorías, evaluaciones.

Aprenda sin volverse peligroso: mejora por evidencia y evaluación, pero siempre con HITL en cambios reales.
